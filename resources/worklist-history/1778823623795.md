## Items proposed

- `multi-file-diff-support` (proposed, `src-tauri/src/lib.rs`)
  - **Before:** Worklist items carry a single `file: <string>` field, used by the /__worklist route to compute the inline diff shown in TO COMMIT rows. Reality is that many feature changes touch 2+ files — core feature was lib.rs + Workspace.xmlui; the framing item is conventions.md + docs/worklist-history.md; track-worklist-history was doc + snapshot folder. The current schema forces the agent to pick one as 'the primary' and describe the others in prose; the inline diff then renders only that one file, leaving the user to verify the rest via terminal git diff or by asking the agent to paste the diff into chat. Surfaced sharply on the framing item: two files touched, inline diff shows one, user reports 'this is pretty confusing.'  Alternatives considered:  (a) Status quo + verify in chat. Agent dumps `git diff` for unrendered files when asked. Cheap but breaks the principle that TO COMMIT review should be self-contained. Rejected because it's the current source of confusion.  (b) Render the full unstaged tree diff per applied item, no path filter. Simple but all applied items would show the SAME diff, breaking per-item attribution when multiple items sit at TO COMMIT simultaneously. Rejected earlier in this session.  (c) Multi-file via a `files: [...]` array, fallback to `file` for backward compat. Each item declares its scope honestly; route fetches and concatenates diffs; the existing line-by-line renderer handles the combined output unchanged (file boundaries are visually distinct via the `diff --git a/...` header lines). Chosen because it expresses item scope correctly without forcing a global rewrite of the schema, and the agent is the one choosing the file list — same prediction burden as today's `file:`, just plural.
  - **After:** Three-file change:  (1) `src-tauri/src/lib.rs` (the /__worklist route handler): replace the single-file diff fetch with a loop over `files` (when present) or `[file]` (backward compat). For each path, run `git diff -- <path>`, falling through to `git diff --no-index -- /dev/null <path>` for untracked files (the existing fallback). Concatenate the resulting diffs with a single newline separator; inject as the item's `diff` field. The Workspace.xmlui renderer needs no change — it splits the diff string on newlines and colors each line by prefix, and the `diff --git a/...` headers between files render as uncolored separator lines that visually delineate per-file sections.  (2) `app/__shell/conventions.md`: update the worklist item schema documentation to mention `files: [...]` as an alternative to `file: <string>`. One-liner about when to prefer it (any change spanning 2+ files).  (3) `resources/worklist.json`: after the route change lands and the binary is rebuilt, edit the in-flight `document-dual-readership-framing` item to use `files: ["docs/worklist-history.md", "app/__shell/conventions.md"]` instead of single `file`, so its TO COMMIT inline diff shows both files together — turning the framing-item's confusion into the demo we wanted.
